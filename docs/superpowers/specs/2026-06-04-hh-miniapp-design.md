# hh Mini App — личный профиль, статистика, тумблеры (MVP)

Дата: 2026-06-04. Статус: одобрено, в реализации.

## Цель
Telegram Mini App: каждый кандидат заходит через своего Telegram, видит **свой** hh-аккаунт
— профиль, статистику и переключатели функций (вкл/выкл). MVP.

## Решения (из брейншторма)
- Хостинг: **домен + Let's Encrypt** на хосте h200. Поддомен `hh.tgbot-afisha.ru → 109.120.183.147`
  (apex/www заняты другим проектом — не трогаем). Порт 80 снаружи открыт (проверено), ufw off, sudo есть.
- Аудитория: **каждый видит свой аккаунт** (маппинг по Telegram user_id).
- Объём v1: профиль + статистика + 4 тумблера. Без детальной конфигурации (лимиты/расписание) — позже.

## Архитектура (вариант A)
- `web_app.py` — FastAPI+uvicorn, новый процесс в ТОМ ЖЕ контейнере (код монтируется томом `.:/app`).
  Слушает `127.0.0.1:60080`. Отдаёт статику Mini App + JSON-API. Ходит в общий Postgres (`public`, по `account`).
- `run_webapp.sh` + cron-watchdog (как `run_listener.sh`) — держит uvicorn живым.
- Хост: **nginx** проксирует `https://hh.tgbot-afisha.ru` → `127.0.0.1:60080`; **certbot** (LE) — TLS.
- Деплой: код — scp+restart (том). Новые зависимости (fastapi/uvicorn) + публикация порта — однократный
  ребилд образа + `docker compose up -d`.

## Авторизация
- Telegram WebApp отдаёт подписанный `initData`. Фронт шлёт его в заголовке каждого запроса.
- Бэкенд валидирует HMAC-SHA256 (секрет — токен бота), проверяет `auth_date` (не старше суток) → `tg_user_id`.
- Маппинг `tg_user_id → account`: `/connect` дописывает `telegram.user_id` в `app_config` аккаунта.
  Бэкенд ищет аккаунт по нему. Нет привязки → экран «привяжи Telegram через /connect».
- Один бот, много пользователей; scope строго по найденному account.

## API (всё под initData-проверкой)
- `GET /api/me` → `{profile, stats}`.
  - profile: имя, hh_id, резюме (title), желаемая зарплата, статус (работает/пауза), последний отклик.
  - stats: отклики (сегодня/всего), просмотры резюме, ответы работодателей, приглашения/интервью,
    воронка `[отклики, просмотры, ответы, интервью]`, тренд за 7 дней.
- `GET /api/settings` → `{apply, reply, browse, giga}` (bool).
- `POST /api/settings` `{key, value}` → переключить (только из белого списка `feat.*`).

## Источники статистики
- hh API (живое): `/resumes/mine` (просмотры), `/negotiations` (кол-во + статусы → ответы/приглашения).
- БД: `notifications`/`action_items` (интервью/хендофф), счётчики откликов, `seen_keys`. Воронка — логика `funnel.py`.
- Короткий кэш в процессе (например 60с), чтобы не дёргать hh на каждый рефреш.

## Тумблеры → реальное отключение
Новые ключи настроек (bool, default `true`): `feat.apply`, `feat.reply`, `feat.browse`, `feat.giga`.
Каждый крон-скрипт в начале (на свой account) делает guard: `if not get_setting('feat.X', True): skip`.
`run_all` уже бежит по аккаунтам. Маппинг:
- `feat.apply` → apply-similar + apply_tests
- `feat.reply` → reply-employers
- `feat.browse` → browse_activity + bump
- `feat.giga` → GigaRecruiter + дайджест/уведомления

## Фронт
Статичные `index.html / app.js / style.css` + Telegram WebApp SDK (`telegram-web-app.js`), тема из Telegram.
Три вкладки: Профиль · Статистика · Функции. Без сборочного тулинга.

## Точка входа
Бот: кнопка меню (menu button) / inline-кнопка `web_app` с URL `https://hh.tgbot-afisha.ru` → открывает Mini App.

## Безопасность
HTTPS-only, валидация+срок initData, запись только ключей из белого списка, CORS только на домен,
бинд uvicorn на 127.0.0.1 (наружу только через nginx), никаких секретов во фронте.

## Тесты
- Юнит: валидация initData (валид/невалид/протух), маппинг tg→account, чтение/запись тумблеров,
  агрегация статистики на моках hh.
- Ручной: открыть из бота, переключить тумблер → убедиться что соответствующий крон-скрипт пропускается.

## Этапы
1. Инфра: deps (fastapi/uvicorn) + порт + skeleton web_app + nginx + certbot → рабочий HTTPS-эндпоинт.
2. Бэкенд: initData-auth + маппинг (правка `/connect`) + `/api/me` + `/api/settings`.
3. Тумблеры: ключи `feat.*` + guard'ы в крон-скриптах.
4. Фронт: 3 вкладки.
5. Кнопка в боте + e2e-проверка.
