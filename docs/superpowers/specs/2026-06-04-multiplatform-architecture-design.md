# Расширяемая архитектура: источники вакансий + обработчики заданий — дизайн (умбрелла)

Дата: 2026-06-04
Статус: проектирование. Это **родительский** спек; детали каждого источника/обработчика — в
отдельных подспеках.

## 1. Контекст

hh-applicant-tool превращается из «бота для hh.ru» в **расширяемую платформу авто-поиска и
авто-отклика** с двумя независимыми осями расширения:

- **Источники вакансий** — откуда берём вакансии и где откликаемся/переписываемся:
  hh.ru, Habr Career, GetMatch, Telegram-каналы (и далее).
- **Обработчики заданий** — что делать с интерактивным заданием, которое прислал
  работодатель/скрининг ПОСЛЕ отклика: пройти интервью у бота (ГигаРекрутер — уже есть),
  заполнить анкету/форму (Google Forms и пр.), пройти тест, и т.д. Их будет много.

Модель идентичности: **кандидат + подключения** (один человек = одна личность; общие
резюме/настройки; единый кабинет; джобы per «кандидат × источник»).

Это **несколько независимых подсистем** — делается по одному подспеку (см. §9). Документ
фиксирует общую архитектуру, в которую они встают.

## 2. Модель данных (целевая)

- **`candidates`** — одна строка на человека: `id`, имя, `resume_text`, `preferences`
  (зарплата, формат работы, фильтры), общие настройки/тумблеры.
- **`connections`** — одна строка на (кандидат × источник): `candidate_id`, `source`,
  креды/сессия (Fernet-encrypted), `config` (jsonb), `enabled`. `source ∈ {hh, habr, getmatch, tg}`.
- Текущий `public.app_users` (hh-аккаунты) → `connections` с `source='hh'`; `candidates` — новая
  сущность. Миграция app_users(hh) — отдельный подспек.
- Тумблеры: **per-candidate** (общие, напр. «Уведомления») и **per-connection** (напр. «Авто-
  отклики на habr») и **per-handler** (напр. «ГигаРекрутер», «Авто-заполнение форм»).

## 3. Ось A — Источники (connectors)

Два вида источников:

1. **Board** (hh, Habr Career, GetMatch): аккаунт → поиск → отклик → переписка. Реализуют общий
   **коннектор-интерфейс** (async, per-источник):
   - `auth() / refresh()`
   - `search_vacancies(filters) -> list[Vacancy]`
   - `apply(vacancy, resume, cover) -> ApplyResult`
   - `list_negotiations() -> list[Chat]`
   - `chat_messages(chat) -> list[Message]` / `reply(chat, text)`
2. **Feed** (Telegram-каналы): мониторинг каналов → парсинг постингов-вакансий → действие.
   Не «аккаунт с откликами», а лента. Через **Telethon-сессию кандидата** (та же, что для
   обработчиков). Контракт: `poll_channels(candidate) -> list[Posting]` + `act(posting)`
   (отклик по ссылке через нужный board-коннектор / ЛС автору). Дедуп через `seen_keys`.

hh — первая реализация board: текущий код прячется за интерфейс (рефактор — отдельный подспек),
после чего общие операции (`apply-similar`, `reply-employers`) вызывают интерфейс, а не hh API
напрямую. habr/getmatch — новые реализации того же контракта.

## 4. Ось B — Обработчики заданий (handlers / processors)

Обработчик = реакция на интерактивное задание от работодателя/скрининга. **Источнико-
агностичны**: одно и то же задание (напр. ссылка на Google Form или на бота-рекрутёра) может
прийти из чата любого источника. Каждый обработчик:

- `detect(task_ctx) -> bool` — по тексту/ссылке/типу понять, что это его задание
  (напр. ссылка `Giga_recruiter_bot?start=`, `docs.google.com/forms`, «пройдите тест по ссылке»).
- `requires` — что нужно (Telethon-сессия / браузер / только LLM).
- `enabled(candidate)` — per-candidate тумблер.
- `handle(candidate, task_ctx)` — выполнить (через Telethon / Playwright / LLM).

**Реестр обработчиков** + единый роутер: входящее задание из любого источника классифицируется
→ если есть **включённый** подходящий обработчик → авто-выполнение; иначе → «Дело» пользователю
(уведомление). Это прямое обобщение текущей связки `notify_actions` (классификатор) +
`giga_recruiter` (обработчик ГР): сейчас «giga активен → не кладём в дела, делаем сами» — станет
общим правилом для любого обработчика.

Обработчики (текущие и будущие):
- **ГигаРекрутер** (есть) — интервью у TG-бота через Telethon.
- **apply_tests** (есть) — браузерные тесты при отклике (Playwright).
- **Google Forms / анкеты** (будущее) — заполнение веб-форм (Playwright + LLM, данные из
  `candidate.resume_text`/`preferences`).
- **Прочие скрининги/тесты** (будущее) — по мере появления.

Оркестрация: каждое семейство-обработчик = операция/джоб (как `giga` сейчас «разгребай очередь
интервью»). Новый обработчик = пакет + JOBS-строка + per-candidate тумблер, **без изменений
оркестратора**.

## 5. Оркестрация

Платформо-обобщённая уже сейчас (подспек/план `prefect-orchestration`): джоб = (источник или
обработчик) × операция на расписании, fan-out по `active_targets(source_or_handler, feature)`.
Новый источник/обработчик = новые JOBS-строки + пакет, без изменений оркестратора.

## 6. Кабинет / статистика

Единый кабинет per-кандидат поверх всех источников: общие резюме/настройки в одном месте;
тумблеры источников и обработчиков; вкладки/фильтр по источнику; агрегированная и
per-источник статистика. Детали — в подспеке кабинета.

## 7. Безопасность / ресурсы

- Креды/сессии — Fernet-encrypted в `connections`.
- Браузерные/LLM-обработчики и источники — под лимитами конкуренции Prefect (`browser`/`llm`),
  чтобы беречь общий сервер с vLLM.

## 8. Что уже есть и куда встаёт

Текущий код — это «источник hh» + два обработчика (ГР, тесты) + классификатор/уведомления,
зашитые под hh. Рефакторинг разносит их по двум осям и за интерфейсы.

## 9. Фазировка (подспеки — по одному, каждый = рабочий инкремент)

1. **Оркестрация на Prefect**, платформо-обобщённая, hh-only — план готов
   (`docs/superpowers/plans/2026-06-04-prefect-orchestration.md`).
2. **Модель данных** `candidates`/`connections` + миграция `app_users(hh)`.
3. **Коннектор-интерфейс источников** + рефактор hh за интерфейс.
4. **Реестр обработчиков** + роутер (обобщение notify_actions+giga); перенос ГР и тестов в реестр.
5. **Habr Career** (source) — коннектор + операции + JOBS.
6. **GetMatch** (source) — коннектор + операции + JOBS.
7. **Telegram-каналы** (source/feed) — коннектор + операции + JOBS.
8. **Google Forms / анкеты** (handler) — обработчик + JOBS.
9. **Единый мульти-платформенный кабинет**.

## 10. Вне области (этого родительского спека)

- Детали реализации каждого коннектора/обработчика/UI — в их подспеках.
- Не меняем бизнес-логику hh-операций здесь (готовим почву интерфейсами — подспеки 3–4).
