# Оркестрация периодических задач на Prefect — дизайн

Дата: 2026-06-04
Статус: проектирование (одобрено направление: Prefect, полная инфра, продуктовый масштаб)
Родитель: [Расширяемая архитектура: источники + обработчики](2026-06-04-multiplatform-architecture-design.md)
— это подспек №1. Оркестрация делается **платформо-обобщённой** (см. §4.5), но сейчас живёт
единственная платформа `hh` (маппится на текущий `app_users`/`HH_ACCOUNT`).

## 1. Контекст и проблема

Сейчас вся периодика hh-applicant-tool крутится в **одном Docker-контейнере** на `cron`:

- crontab (том `/app/crontab`, перезагрузка через `startup.sh` по `@reboot`) запускает задачи:
  `update-resumes` (2ч), `browse_activity` (2ч), `apply-similar` (час), `notify_actions` (:20),
  `reply-employers --use-ai` (:30), `apply_tests` (3ч), `giga_recruiter` (3 мин),
  `refresh-token` (1 мин), `monitor` (день 08:00 МСК), `funnel` (день 08:05), `send_digest` (:15/:45).
- `run_all.py` для каждой задачи **последовательно** проходит по активным аккаунтам из
  `public.app_users` и запускает subprocess с `HH_ACCOUNT=<account>`.
- Анти-бот джиттер — `sleep $((RANDOM % N))` в каждой cron-строке.
- Перекрытия гасятся **Postgres advisory-локами** (giga).
- Долгоживущие сервисы (uvicorn Mini App, `tg_connect_bot` /connect-листенер) держатся
  **pgrep-вотчдогами** (раз в минуту: не запущен → поднять).
- Логи — один общий `/var/log/cron.log` (часть скриптов без `-u` буферят вывод).

### Боли
1. **Наблюдаемость**: только grep по общему логу; нет истории ранов, статусов, длительностей,
   логов по задаче/аккаунту; буферизация скрывает прогресс.
2. **Надёжность**: best-effort, нет ретраев и алертов о падениях; тихие сбои.
3. **Управление**: добавить/поменять задачу = править текст crontab + редеплой; интервалы
   зашиты в часы UTC.
4. **Инженерная чистота**: расписание размазано по crontab + sleep-джиттеру + feat-гейтам в
   скриптах + advisory-локам; вотчдоги через pgrep; fan-out последовательный (медленный
   аккаунт блокирует остальных).

## 2. Цели

Все четыре сразу (подтверждено):
- **Наблюдаемость / дашборд** — единое место: что/когда запускалось, success/fail, длительность,
  логи по задаче и аккаунту.
- **Надёжность** — ретраи, алерты о падении/непуске, корректные локи вместо advisory-хаков и pgrep.
- **Удобство управления** — вкл/выкл и параметры per-account из кабинета/конфига без правки crontab.
- **Правильный фреймворк + UI** — заменить cron нормальным оркестратором.

Масштаб: продуктовый (десятки-сотни аккаунтов), горизонтальное масштабирование воркеров.
Инфра: полная (отдельные контейнеры) допустима. Сервер общий с vLLM — **vLLM не трогаем**,
бережём его ресурсы лимитами конкуренции.

## 3. Выбор: Prefect

Выбран **Prefect 3** (self-hosted). Причины: Python-native (наши скрипты заворачиваются в
`@task`/`@flow` почти как есть), красивый UI из коробки, встроенные ретраи / расписания /
лимиты конкуренции, work-pools для горизонтального масштаба, бэкенд — внешний Postgres
(уже есть). Прямое попадание во все 4 цели и самая дешёвая миграция с текущих скриптов.

Рассмотренные альтернативы: **Dagster** (изящные партиции-по-аккаунтам и сенсоры, но больше
концепций и тяжелее), **Celery+Beat+Flower** (максимальная пропускная способность, но слабый UI
и больше ручного клея). Зафиксированы как возможные на случай смены приоритетов.

## 4. Целевая архитектура

### 4.1 Контейнеры (docker-compose; код — bind-mount как сейчас)
```
postgres        (есть)   — данные приложения + отдельная БД `prefect` под метаданные Prefect
vllm            (есть)   — НЕ трогаем
prefect-server           — Prefect API + UI (дашборд); PREFECT_API_DATABASE_CONNECTION_URL -> postgres/prefect
prefect-worker  ×N       — тянут раны из work-pool'ов, исполняют флоу; масштаб = +воркер
hh-web                   — uvicorn web_app:app, restart: always  (заменяет uvicorn-вотчдог)
hh-listener              — tg_connect_bot.py, restart: always    (заменяет listener-вотчдог)
```
Долгоживущие сервисы (web, listener) выводятся из «оркестрации» в обычные сервисы с
авто-рестартом. Оба pgrep-вотчдога удаляются.

Work-pools (раздельные очереди под разную нагрузку):
- `default` — лёгкие задачи (apply, reply, notify, digest, browse, update-resumes, refresh-token, monitor, funnel).
- `browser` — `apply_tests` (Playwright), лимит конкуренции 1–2.
- `giga` — разгребание очереди интервью, лимит конкуренции per-account = 1.

### 4.2 Модель «диспетчер → per-account» 
Каждая задача — flow-диспетчер на расписании, который читает активные аккаунты и раздаёт
работу per-account задачам:
```
@flow(name="apply-similar")
def apply_similar():
    for account in active_accounts():          # из app_users
        apply_one.submit(account)              # параллельно, с лимитом конкуренции

@task(retries=2, retry_delay_seconds=[30, 120], tags=["apply", "acct:{account}"])
def apply_one(account: str):
    with account_ctx(account):                 # выставляет HH_ACCOUNT-контекст
        run_existing_apply_similar()           # ВЫЗОВ существующего кода операции
```
- Тело per-account задачи — **существующий код** (`apply-similar`, `reply-employers`,
  `notify_actions`, `send_digest`, `browse_activity`, `apply_tests`, `update-resumes`,
  `refresh-token`, `monitor`, `funnel`, `giga`). Оборачиваем, не переписываем.
- Fan-out параллельный (с лимитом), не последовательный как в `run_all`. Падение одного
  аккаунта изолировано (свой ран, свои ретраи), остальные идут.
- `active_accounts()` фильтрует по feat-флагам и подключению (как сейчас гейтят скрипты).

### 4.3 Расписания (deployments)
Деплойменты с cron/interval, рабочее окно 08:00–22:00 МСК (05–19 UTC) сохраняем. Маппинг 1:1
с текущим crontab. Расписание — в коде (версионируется), не в тексте crontab.
Анти-бот джиттер — случайная задержка в начале флоу (или schedule jitter Prefect).

### 4.4 giga и «очередь-воркер»
`giga` — flow на коротком интервале (как сейчас ~3 мин), per-account задача с **лимитом
конкуренции `giga:<account>=1`**: это заменяет advisory-лок и делает перекрытия невозможными.
Логику continuous-session (StartBot, оценка 5★, авто-очередь, чистка «дел») не трогаем —
переносим тело как есть. (Переход на событийный триггер/сенсор — возможная будущая итерация,
не сейчас.)

### 4.5 Платформо-обобщённость (фундамент под источники/обработчики)

Оркестратор не знает про hh — он знает «запусти задачу J платформы P по её активным целям на
расписании S». Заложено сразу, чтобы Habr/GetMatch/TG/обработчики втыкались без его изменений:

- В JOBS-таблице есть поле **`platform`** (сейчас у всех `hh`; потом `habr`/`getmatch`/`tg`/
  имя обработчика).
- Fan-out — через **`active_targets(platform, feature)`**: для `hh` это текущие `app_users`;
  для других — кандидаты с активным подключением на этой платформе и включённой фичей.
- Контекст сабпроцесса — обобщённый: маппинг `platform -> env` (`hh -> HH_ACCOUNT`); цель
  передаётся в нужную переменную.
- Имя деплоймента/флоу — `"<platform>-<job>"` (`hh-apply-similar`, далее `habr-apply`,
  `tg-channels-scan`, …).

Новая платформа/обработчик = новые JOBS-строки + свой пакет операций. Бизнес-логику и модель
`candidates/connections` определяют родительский спек и его подспеки — здесь только оркестрация.

## 5. Надёжность
- **Ретраи** на per-account задачах (транзиентные ошибки hh/LLM): `retries` + backoff.
- **Лимиты конкуренции** (Prefect concurrency limits / tags) вместо advisory-локов и глобальные
  кэпы для защиты vLLM: LLM-тяжёлые и браузерные флоу — 1–2 параллельно.
- **Алерты**: Prefect Automation на `Failed/Crashed` флоу-ран → уведомление в Telegram через наш
  существующий `pgconn.notify()` (или Prefect notification block). `monitor` dead-man-switch
  оставляем как второй контур.
- **Сервисы**: web/listener авто-рестартятся политикой compose; pgrep-вотчдоги удаляются.

## 6. Наблюдаемость
- **Prefect UI** — основной дашборд: флоу-раны, таск-раны, состояния (success/failed/retrying/
  crashed), длительности, **логи по каждому рану**, per-account через параметры/теги, календарь
  расписаний, ближайшие запуски.
- **Кабинет (для админа)** — компактный виджет «здоровье задач» (последние раны/статусы),
  читающий Prefect API. Опционально на первом этапе.
- Структурированные логи в Prefect заменяют единый cron.log.

## 7. Управление
- Вкл/выкл и параметры per-account (`feat.*`, `apply.max_per_day`, лимиты, интервалы) остаются в
  Postgres-настройках; диспетчер читает их **каждый ран** → правка в кабинете действует со
  следующего запуска, без правки crontab и редеплоя.
- Новая задача = новый `@flow` + deployment в коде (а не строка в crontab).

## 8. Безопасность
- Prefect UI/API за nginx + авторизацией (как Mini App): bind 127.0.0.1, наружу через TLS-домен,
  не публично.
- API-ключ между worker↔server.
- Секреты (hh-токены, telegram-токен, Telethon-сессии) — в Postgres / Prefect blocks, не в коде.

## 9. План миграции (поэтапно, без big-bang)
1. Поднять `prefect-server` + `prefect-worker` + БД `prefect` рядом с **живым cron** (параллельно).
2. Перевести сервисы: `hh-web` и `hh-listener` → compose-сервисы `restart: always`; удалить их
   pgrep-вотчдоги из crontab.
3. Портировать задачи **по одной**: реализовать flow+deployment, прогнать параллельно/в dry,
   сверить вывод с cron в Prefect UI, затем удалить соответствующую cron-строку.
4. Когда все задачи перенесены и сверены — снести crontab, `run_all.py` fan-out (его заменяет
   диспетчер) и код advisory-локов (его заменяют лимиты конкуренции).
5. Финал: cron-роль уходит полностью; вся периодика — в Prefect.

Откат на каждом шаге: вернуть cron-строку (старый код не удаляется до полного паритета).

## 10. Тестирование
- Тело флоу = существующая операция (запускается standalone) → смоук через
  `prefect deployment run <name>` + наблюдение в UI.
- Dry-режимы операций сохраняем для безопасной сверки.
- Parity: старый cron-вывод vs новый флоу-ран на одинаковых данных.
- Изоляция: искусственно роняем один аккаунт — остальные per-account раны проходят.

## 11. Вне области (этой итерации)
- Не меняем бизнес-логику операций (apply/reply/giga/...). Только оркестрацию вокруг них.
- Не трогаем vLLM и схему данных приложения (кроме отдельной БД `prefect`).
- Кабинет-виджет «здоровье задач» — опционально, может быть отдельной итерацией.

## 12. Решения, требующие подтверждения
- БД метаданных Prefect: **отдельная логическая БД `prefect` на существующем Postgres**
  (рекомендация — без ещё одного контейнера), либо отдельный Postgres-инстанс.
- Версия: **Prefect 3** (актуальная).
- Экспонирование UI: поддомен + Let's Encrypt + basic-auth (как Mini App).
