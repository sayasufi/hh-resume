# Источник GetMatch — авто-отклик через Telegram-бота @g_jobbot — дизайн

Дата: 2026-06-05
Статус: проектирование (подспек §9.6 умбреллы `2026-06-04-multiplatform-architecture-design.md`).
Это **новый источник вакансий** — шестая фаза умбреллы, делается **standalone** (как giga),
без блокировки на рефакторе коннектор-интерфейса (фазы 3-4). Интерфейс извлечём позже.

## 1. Контекст и модель GetMatch (по живой разведке)

GetMatch (getmatch.ru) — IT-площадка, центр взаимодействия кандидата = **Telegram-бот
[@g_jobbot](https://t.me/g_jobbot)**. Механика (изучена вживую через сессию кандидата):

- Бот шлёт **ленту вакансий**, предотобранную под параметры кандидата. Каждая вакансия =
  сообщение: заголовок-ссылка `https://getmatch.ru/vacancies/{id}-{slug}?...` + inline-кнопки:
  **`💥 Откликнуться в боте`** | `Описание` / `Вакансии (NNN)`. Внизу пачки — `Ещё N вакансий для вас`.
- **Отклик = ОДИН клик** по `💥 Откликнуться в боте` → бот: «Отклик отправлен работодателю».
  **Сопроводительное письмо НЕ требуется.** Есть `/discard_<token>` (отменить).
- **Фильтры — на стороне GetMatch** (`/settings`): Специальность, Регион и формат работы,
  Зарплата (порог). Лента уже отфильтрована — **свой фильтр не нужен.**
- **Профиль** (`/profile`): состояние «подтверждённый профиль 🤘 — можно откликаться в один
  клик». Отклик в один клик доступен ТОЛЬКО при подтверждённом профиле.
- **Отклики** (`/applications` → «📑 Мои отклики»): статусы, контакты рекрутёра. Прямого
  чата «работодатель↔кандидат» как negotiations в hh НЕТ — посредничают HR GetMatch.
- Прочие команды: `/job_offers` (смотреть вакансии), `/profile_up`/`/open_profile` (заметность),
  `/salaries` (справочник зарплат), `/human_support` (эксперт-человек).

Вывод: GetMatch = **«лента + отклик в один клик»**. Это паттерн ГигаРекрутера (Telethon + клик
по кнопке), только проще — без писем, без переписки, без своего фильтра.

## 2. Цель

Автоматически откликаться от лица кандидата на **новые** вакансии из ленты @g_jobbot (один клик,
дедуп, дневной лимит, человеко-подобно), **только при полноценном подтверждённом профиле**.
Гейтинг и кабинет — как у giga. Без писем/переписки/своего фильтра в v1.

## 3. Архитектура

Новый источник = одна операция-джоб, переиспользующая Telethon-инфраструктуру (та же
зашифрованная user-сессия кандидата `app_config.tg_user_session`, тот же `pgconn`, тот же
паттерн advisory-лока и `seen_keys`, что у `giga_recruiter`).

- **`services/getmatch_apply.py`** — операция (новый файл, зеркало структуры `giga_recruiter.py`).
- Гейтинг: тумблер **`feat.getmatch`** (требует `tg_user_session`, как giga) +
  `active_targets("getmatch", "getmatch")` для fan-out оркестратором.
- Оркестрация: **JOBS-строка `getmatch`** в `orchestration/flows.py` — без изменений оркестратора.
- Кабинет: тумблер «GetMatch» + статистика откликов через GetMatch.

Ничего в ядре `hh_applicant_tool` и в оркестраторе менять не требуется — только новый пакет/файл,
строка JOBS, тумблер, кусок кабинета.

## 4. Операция `getmatch_apply.py` — поведение

Аргумент `--dry` (ничего не кликает, только логирует намерение). Один запуск на аккаунт:

1. **Single-flight**: Postgres advisory-лок по `(account)` (как giga). Если уже выполняется — выход.
2. **Гейт фичи**: `feat.getmatch` выключен или нет `tg_user_session` → выход.
3. **Telethon-коннект** сессией кандидата; `is_user_authorized()` — иначе уведомление «переподключи
   Telegram» и выход.
4. **ПРОВЕРКА ПРОФИЛЯ (обязательная)**: отправить `/profile`, прочитать ответ. Профиль считается
   полноценным, если в ответе есть маркер подтверждённости (regex `подтвержд` И/ИЛИ «в один клик»).
   - **Не подтверждён** → НЕ откликаемся. Кладём `pgconn.notify(PRIORITY_MED, …, category="action")`
     + `add_action_items` дело: «Заполните/подтвердите профиль на GetMatch (@g_jobbot, /profile),
     чтобы бот начал откликаться». `dedup_key="getmatch:profile"`. Выход.
   - **Подтверждён** → продолжаем.
5. **Получить ленту**: отправить `/job_offers` (или прочитать свежие push-сообщения ленты).
   Точный способ выборки (через `/job_offers` против чтения push-ленты против клика
   `Ещё N вакансий`) уточняется коротким спайком в реализации; дизайн предполагает: триггерим ленту
   → обрабатываем сообщения-вакансии → пагинация кнопкой `Ещё N вакансий`.
6. **Цикл откликов** (до `getmatch.max_per_day` или конца ленты):
   - Для каждого сообщения-вакансии:
     - Извлечь `vacancy_id` из URL `getmatch.ru/vacancies/{id}-…` (regex).
     - Если `id` уже в `seen_keys("getmatch")` → пропустить.
     - Найти кнопку отклика **защитно**: первая inline-кнопка, чей текст содержит «Откликнуться»
       (или эмодзи 💥). Нет кнопки (уже откликнулись / профиль не даёт один клик) → пропустить,
       пометить seen.
     - `--dry`: логировать «откликнулся бы», пометить seen, НЕ кликать.
     - Иначе: `message.click(<кнопка>)`; подтверждение успеха по ответу бота (regex
       `отклик.{0,15}отправлен` / `📑 Мои отклики`).
     - `add_seen("getmatch", [id])`, `bump_activity("getmatch", 1)`, счётчик++, человеко-пауза
       (`random.uniform(5, 15)`с).
     - Достигнут дневной лимит → стоп.
   - Лента кончилась/нет новых → клик `Ещё N вакансий` для следующей пачки; нет кнопки → конец.
7. Печать сводки: проверено вакансий, откликов отправлено, лимит.

**Дневной лимит**: `settings.getmatch.max_per_day` (дефолт 50). Дневной счётчик — `bump_activity`/
`activity_daily(kind="getmatch", day=today)`; перед циклом читаем сегодняшнее значение, не превышаем.

## 5. Данные

- **`seen_keys`** kind=`"getmatch"`, key = `vacancy_id` (строка). Дедуп откликов навсегда.
- **`activity_daily`** kind=`"getmatch"` — дневной счётчик откликов (для лимита и кабинета).
- **`settings`**: `getmatch.max_per_day` (int, дефолт 20).
- **`feat.getmatch`** — тумблер (как остальные feat.*).
- **`action_items`**: дело «подтвердить профиль» при неполном профиле (dedup_key `getmatch:profile`).
- Никаких новых таблиц — всё в существующих (`seen_keys`, `activity_daily`, `settings`, `action_items`).

## 6. Гейтинг и оркестрация

- `web_app.FEATURES` += `"getmatch"`. `_set_config`: `feat.getmatch=True` запрещён без
  `tg_user_session` (как giga, HTTP 400 с подсказкой).
- `orchestration/flows.py` JOBS += строка:
  `dict(name="getmatch", command=["python","/app/services/getmatch_apply.py"], feature="getmatch",
   cron="10 6-18/4 * * *", jitter=120, tags=[], timeout=1200)` — несколько раз в день в рабочем
  окне, лёгкая (Telethon, без llm/browser-лимитов). Точный cron — на ревью.
- `active_targets("getmatch","getmatch")` отберёт кандидатов с включённой фичей.

## 7. Кабинет (Mini App)

- **Тумблер «GetMatch»** в разделе «Прочее»/«Источники» (рядом с ГР): `data-feat="getmatch"`,
  блокируется без `tg_connected` (как giga), с подсказкой «нужен подключённый Telegram».
- **Статистика**: в «Активность бота» добавить счётчик «Откликов через GetMatch»
  (`activity_daily kind=getmatch`); опционально мини-карточка «GetMatch: N откликов сегодня».
- (Опц., позже) показать дело «подтвердить профиль», если профиль неполный.

## 8. Безопасность и устойчивость

- **Profile-гейт** (§4.4) — НЕ откликаемся с неполным профилем (требование заказчика).
- **Advisory-лок** per-account — без гонок/двойных откликов.
- **Защитный матч кнопки** по подстроке «Откликнуться»/эмодзи — устойчиво к смене текста ботом.
- **Дневной cap + паузы** `random.uniform(5,15)`с — человеко-подобно, не триггерим антифрод.
- **`--dry`** — безопасный прогон без реальных откликов (в отличие от старого reply: тут dry
  действительно ничего не кликает).
- Кнопка/ответ не найдены, бот молчит, лимит флуда Telegram → логируем и мягко выходим, не падаем.
- Не трогаем `/discard`, `/profile_up`, `/open_profile`, `/human_support` — только чтение профиля
  и клик отклика.

## 9. Тестирование

Чистые хелперы — юнит-тестами (`tests/test_getmatch.py`):
- `extract_vacancy_id(url)` — из разных форматов URL `getmatch.ru/vacancies/34650-…`.
- `is_profile_ok(text)` — маркеры подтверждённого профиля / неполного.
- `find_apply_button(buttons)` — защитный матч по подстроке/эмодзи.
- `applied_ok(text)` — подтверждение успешного отклика.
- Логика дневного лимита/дедупа (на фейковых seen/activity).
Telethon-взаимодействие (сеть) — не юнит-тестим; проверяется `--dry` на живой сессии + ручной спайк.

## 10. План внедрения (этапы)

1. Спайк: подтвердить способ выборки ленты (`/job_offers` vs push vs `Ещё N`) + точные regex
   маркеров (профиль, успех отклика) на живой сессии Семёна, `--dry`.
2. `getmatch_apply.py` + хелперы + тесты.
3. Гейтинг (`feat.getmatch`, web_app), JOBS-строка, тумблер в кабинете, счётчик активности.
4. Прогон `--dry` на Семёне → проверка выбора кнопок/вакансий без реальных откликов.
5. Боевой прогон с маленьким лимитом (напр. 3) → проверить реальные отклики + дедуп + «Мои отклики».
6. Поднять лимит, поставить на расписание.

## 11. Вне области v1

- Свой фильтр (зарплата/формат/ГПХ) — GetMatch фильтрует сам через `/settings`. (v2 — опц.)
- Чтение «Мои отклики»/статусов в кабинет — в v1 показываем свой счётчик откликов.
- Веб/Playwright и приватный API GetMatch — отвергнуты (бот проще и официальнее).
- Извлечение коннектор-интерфейса источников — отдельная фаза умбреллы (3), не блокирует GetMatch.
